       - `PipelineMapData` interface: `{ pipeline_id: string; pipeline_name: string; stages: Array<CjmPipelineFlowRow & { clients: CjmClientPosition[] }> }`
       - `CjmClientPosition` interface: `{ contact_id: string; contact_name: string; pipeline_id: string; current_stage: string; entered_stage_at: string; sla_status: CjmSlaStatus; hours_in_stage: number }`
       - `CjmTab` type: `"map" | "analytics" | "sla" | "editor"`

    2. **Create `useCjmPipelineMap.ts`**:
       - Fetch from `vw_cjm_pipeline_flow` view (SELECT * WHERE location_id = selectedAccount?.location_id)
       - Group rows by `pipeline_id` into `PipelineMapData[]` — each pipeline has array of stages sorted by `stage_order`
       - Return `{ pipelines, loading, error, refetch }`
       - Use `supabase` from `../lib/supabase` (same pattern as existing hooks like `useDashboardMetrics`)
       - Accept optional `locationId: string` param (use `selectedAccount?.location_id` pattern from existing hooks, check how `useClientCosts` or similar hook gets the location)

    3. **Create `useCjmClientPositions.ts`**:
       - Fetch from `cjm_journey_state` WHERE location_id = param
       - For contact_name: query `cjm_events` metadata->>'contact_name' for each contact_id (use a subselect or a second query). Fallback: use `contact_id` substring if name not in metadata.
       - Compute `hours_in_stage` client-side: `(Date.now() - new Date(entered_stage_at).getTime()) / 3600000`
       - Return `{ positions: CjmClientPosition[], loading, error, refetch }`

    4. **Create `useCjmStageConfig.ts`**:
       - Fetch all stage configs WHERE location_id = param, ordered by pipeline_id + stage_order
       - `updateStageConfig(id: string, changes: Partial<CjmStageConfig>)` — optimistic update pattern: update local state immediately, then `supabase.from('cjm_stage_config').update({...changes, updated_at: new Date().toISOString()}).eq('id', id)`. On error, refetch and show error via console.error (toast integration comes in Plan 10-02).
       - Return `{ stageConfigs, updateStageConfig, loading, error, refetch }`

    5. **Create `useCjmRealtime.ts`**:
       - Subscribe to Broadcast channel `cjm-events`, event `stage_change`
       - On event, call `onStageChange(payload)` callback (passed as arg)
       - Use `useRef` for callback (avoid re-creating channel on callback change) — follow pattern from research
       - Also include 30s polling fallback: `setInterval(refetch, 30000)` where `refetch` is passed as second arg. This ensures map updates even if Broadcast node is not yet added to n8n Event Ingester.
       - Cleanup: `supabase.removeChannel` + `clearInterval` on unmount
       - Return `{ isSubscribed: boolean }`

    6. **Update `hooks/index.ts`** — Add 4 new exports:
       ```
       export * from './useCjmPipelineMap';
       export * from './useCjmClientPositions';
       export * from './useCjmStageConfig';
       export * from './useCjmRealtime';
       ```

    7. **Create SQL file** `.planning/phases/10-visual-journey-map/sql/10-rls-update-policy.sql`:
       ```sql
       -- Phase 10: Allow authenticated users to update stage configs
       DROP POLICY IF EXISTS "auth update cjm_stage_config" ON cjm_stage_config;
       CREATE POLICY "auth update cjm_stage_config"
         ON cjm_stage_config FOR UPDATE TO authenticated
         USING (true) WITH CHECK (true);
       ```
       NOTE: This SQL must be applied manually by the user via Supabase SQL Editor before inline editing will work.

    IMPORTANT: Use `supabase` import from `../lib/supabase` (NOT `../../supabaseClient.ts` which is legacy). Follow the same hook structure as existing hooks (useState for data/loading/error, useEffect for fetch, return object).
